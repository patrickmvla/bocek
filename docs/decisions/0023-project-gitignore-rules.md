# ADR-0023: Project .bocek/ Gitignore Rules

## Status
Accepted

## Context
The `.bocek/` directory in a project contains both permanent knowledge (vault entries that should travel with the code) and transient state (current mode, compiled artifacts). The gitignore rules determine what gets committed and what stays local.

## Decision

**Gitignored (not committed):**
- `.bocek/vault/.compiled/` — build artifact, regenerated by implementation primitive on demand
- `.bocek/mode` — per-developer state (which primitive is active)

**Committed:**
- `.bocek/vault/index.md` — vault entry point and navigation
- `.bocek/vault/**/*.md` (excluding .compiled/) — all vault entries (decisions, research, contracts, context)
- `.bocek/state.md` — current exploration state (committed for v1 solo use; revisit for team workflows)

The vault is the product — it travels with the code. Every decision, research finding, and contract is version-controlled alongside the codebase it describes.

## Alternatives Considered
**Gitignore all of `.bocek/`:** Would make the vault local-only. Rejected — the vault's value is that it persists across machines, sessions, and time. A vault that doesn't travel with the code can't inform future development.

**Gitignore `state.md`:** Would lose session continuity across machines. Accepted for team workflows (post-v1) where different developers have different active work. For v1 solo use, committing it provides cross-machine continuity.

## Consequences
- **Positive**: Vault knowledge travels with the code — clone the repo, get the design history
- **Positive**: Compiled artifacts don't pollute git history
- **Positive**: Mode file is per-developer — no conflicts from different developers in different modes
- **Negative**: Vault entries appear in diffs and PRs — but this is a feature, not a bug (vault changes should be reviewed)

## Revisit When
- Team workflows are designed — `state.md` should likely be gitignored in multi-developer projects
